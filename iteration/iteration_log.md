# Skill 迭代升级记录

## 迭代 1: 实时行程调整 + 天气感知 + 飞书适配

**日期**: 2026-07-12
**版本**: v1.0.0 → v1.1.0

### 痛点描述

初始版本（v1.0.0）存在三个核心问题：

1. **行程是"死"的**：用户在旅途中遇到突发情况（下雨/累了/想吃某种东西），原有行程完全作废，只能重新规划，无法对剩余天数做增量调整。
2. **缺乏天气感知**：没有收集出发日期，无法查询实际天气。雨天的行程和晴天一样全是户外景点，体验极差。
3. **飞书体验冗余**：在飞书上使用时，Agent 的对话式引导过于冗长，用户只想看到最终卡片，不需要中间的多轮寒暄。

### 量化影响

- 行程调整需要完整重新对话，耗时 5-8 轮
- 雨天的行程适用性为 0（全是户外景点无法执行）
- 飞书用户需要阅读大量非必要对话文本才能拿到最终结果

### 假设原因

- SKILL.md 缺少"中途调整"的工作流定义
- 信息收集流程缺少"出发日期"这一关键维度
- 没有区分飞书和桌面端的交互模式

### 实现方案

1. **新增 Step 2: 收集出发日期**
   - 支持多种日期格式解析（"7月20日"、"下周五"、"7/20"）
   - 计算距出发天数，决定天气策略

2. **新增天气处理模块**
   - 15天内 → 查询天气网站（wttr.in / weather.com.cn）
   - 15天外 → 生成 A/B 双方案（晴天版 / 雨天版）
   - 极端天气 → 主动建议改期，推荐最近的好天气日

3. **新增 Platform Awareness 模块**
   - 飞书模式：提问精简为单行，最终仅输出卡片
   - 桌面模式：保持自然对话，输出完整 Markdown

4. **新增 Real-Time Mid-Trip Adjustment 模块**
   - 定义 7 种触发词：累了/下雨了/想吃X/不想去X/还有时间/太贵了/改一下明天
   - 每种触发词有明确的 agent 行为映射
   - 只调整当前及未来天数，已完成天数不动

### 预期效果

- 用户中途调整只需 1 轮对话
- 天气感知让雨天行程有可执行的室内方案
- 飞书用户体验：3 条简短问题 + 1 张卡片，零冗余

## 迭代 2 (计划中)

### 预期痛点

用户每次都要重复说"我不喜欢爬山"等偏好，系统缺乏长期记忆。

### 假设原因

SKILL.md 未利用 Hermes 的 memory 工具持久化用户偏好。

### 计划方案

- 每次行程结束后，自动将偏好关键词写入 memory
- 下次规划时，从 memory 注入历史偏好作为默认约束
- 用户可以用"还是老样子"跳过偏好输入步骤
